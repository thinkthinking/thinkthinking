# 模式 B · 功能模块 PRD 写作指南

> **产出文件：** `.context/vibe-product-design/<功能名 slug>/PRD.md`（单文档，不拆章、不需要合并脚本）
> **输入：** 已有产品背景（若存在模式 A 的 `PRD.md`/分章文件，直接读取引用） + 轻量问卷澄清结论 +
> 高保真原型图（平台与风格由配置驱动，默认 PC 端 2K ＋ 苹果设计风格，存于同目录 `assets/`）。
> **作用：** 让团队里的每个角色，拿着**这一份文档**就能各自开工——不需要再追问你、不需要再开会对齐。
> **写作节奏（PRD ⇄ 原型图 的螺旋迭代）：** 先写初稿把业务逻辑想清楚（4.2 原型图先留空）→ 依初稿
> 确定图单 → 出图 → **图文互检**（攒「设计发现清单」：漏了什么/想错什么/新灵感/图文不一致 →
> 逐条修订进对应章节 → 补图注）→ 修订若改变界面设计则补画/重画受影响的图、再对一遍 → 收敛后格式
> 规范化。画图是一次设计推演，图出来后 PRD 必须跟着变得更完整；文档和原型互为草稿、互相打磨，
> 交付一对互相印证的 PRD ＋ 原型图——这是**设计输入版**。设计师据此完成正式设计稿后，用户带着
> 定稿切图回来，走**设计稿回填**（阅读切图 → 攒差异清单 → 回填全文 → 4.2 切换为定稿切图）形成
> **终版 PRD**——始终是同一份 `PRD.md`，AI 原型图留存进"存档"小节不删除。详细步骤见 `SKILL.md`
> 「模式 B · 执行步骤」，自查方法见下方「图文互检的自查清单」与「设计稿回填：从定稿切图到终版 PRD」。

## 写给谁看，这份 PRD 就该让谁一眼看懂

这是本模板最重要的检验标准，写完自查一遍：

| 角色 | 该看哪一章 | 看完应该能回答的问题 |
| --- | --- | --- |
| UI/UX 设计师 | 第 4 章 | 每个界面/状态长什么样？关键交互怎么走？要复用哪些现有组件？ |
| 研发 | 第 5、6 章 | 业务规则是什么？状态怎么流转？异常怎么处理？要交付哪些功能点？ |
| 测试 | 第 7 章 | 要测哪些边界值、异常输入、权限组合？验收标准是什么？ |
| 产品经理 | 全篇，重点第 1、6 章 | 目标达成没有？范围有没有跑偏？能不能验收？ |

如果某一章的读者看完还要来问你"那具体是什么意思"，这一章就没写到位。

## 写作要求

- **收口，不做全量分析。** 不要重新分析用户画像、竞品、商业模式——那是模式 A 的事。这里只服务
  **这一个功能点**：它解决什么问题、怎么设计、怎么运转、边界在哪。
- **像造火箭说明书一样精确。** "优化交互体验"是废话；"提交按钮在表单未填完时置灰，鼠标悬停提示
  缺失字段"才是内容。每条业务规则、每个异常分支都要具体到可以直接转成判断条件或测试用例。
- **只讲产品需求，不做技术方案设计。** 讲清楚业务规则、状态流转、异常时用户/系统的行为——不负责
  设计数据库表结构、接口协议、技术选型；时序图和状态机画的是"谁在什么条件下做了什么"，不是技术
  实现方案。这份文档要留给设计师、研发、测试各自去做专业判断，PM 负责把"要什么、为什么、边界在哪"
  定清楚。
- **必画异常路径。** 核心流程图和业务规则里，每一步都问"这一步失败/超时/被拒绝会怎样"——这是
  和模式 A 第 04 章一致的原则，此处同样适用，不能省。
- **可追溯、不臆造背景。** 涉及现有产品的部分（已有页面、已有组件、已有数据）要基于用户提供的真实
  信息或已有 PRD 文件；不确定就问用户，不要编。外部事实（技术基准、合规要求等）仍遵循 SKILL.md
  的「联网核实」原则。
- **优先级/工作量/编码复用现成规范。** 见 `输出格式规范.md` 的 P0–P3、XXS–XXL、Low–Very High
  分级，不要重新发明一套。
- **原型图必须配文字说明。** 图是高保真视觉稿，但仍只是辅助：图注要把关键交互点、状态差异、关键设计
  决策讲清楚——设计师和研发都可能看不到图（渲染失败/离线阅读），文字要能独立说清楚逻辑。

## 6. 功能清单与验收标准
优先级、工作量、技术复杂度分级规范见 `输出格式规范.md`（沿用同一套 P0–P3/XXS–XXL/Low–Very High）。

- 若这个功能挂在某个已有模式 A 全量 PRD 之下、且那份 PRD 里已经给这块业务定过 L1 模块码（如

- 每个 ID 对应一条可独立验收的用户故事，不需要再往下拆 L2/L3/L4——这一份 PRD 本身就是最小颗粒度。

| ID | 描述（用户故事） | 优先级 | 工作量 | 验收标准 |
|---|---|---|---|---|
| ANC-01 | 作为…我想要…以便于… | P0 | S | Given <前提> When <操作> Then <预期结果> |

验收标准统一用 **Given-When-Then**（给定前提 - 触发操作 - 预期结果）格式，方便测试直接转成用例，
也方便 PM 验收时逐条打钩。

## 原型图生成规范（config 驱动平台与风格 · 用 `zenmux-image-generation` 出高保真界面稿）

产出的是**高保真界面稿**，把每个关键界面最重要的设计决策展现出来——不是灰框线框图。

**平台与风格都不写死在这份文档里，而是由配置驱动**。用户只填两个字段：

- **`platform`（目标平台，强约束）**：只选一个（默认 `web`，即 PC 端 2K）。本功能**所有页面级
  原型图**的分辨率（`size`）、画幅与界面形态（布局范式、导航结构、信息密度）一律按这个平台出——
  配置了 PC 端，就全部出 PC 端的图，不要有的图画成手机端；仅组件/局部特写这类非页面级配图可临时
  用其它尺寸（如 1024x1024 方形），风格仍须一致。
- **`active_theme`（风格主题）**：一句话（默认苹果设计风格）。skill 通过一步**预处理**把这句主题
  用大模型展开成一份可复用的「风格提示词文件」并缓存，出图时**加载**这段风格提示词、拼在每张图的
  具体画面描述之前。用户不需要手填配色/字体/留白等一堆细节。

**配置存放在用户项目的 `.context` 下，不在 skill 目录里**——`style_prompt.ts` 首次运行会把 skill
的配置模板复制到 `<project-dir>/.context/vibe-product-design/config/`（只在缺失时复制，不覆盖用户
改动），之后读写都在这里，这样 skill 升级覆盖自己的文件时不会清掉用户配置与缓存。

```text
prototype-style.yaml（.context 下）：platform（目标平台）＋ active_theme（一句话风格主题）
  → 预处理：style_prompt.ts plan → 若无缓存，按 brief 用大模型展开成 .context/.../generated/<slug>.style.md
  → 加载：style_prompt.ts load → 取出风格提示词前缀 ＋ rendering（model/quality/size，size 已按 platform 定死）
  → 每张图 prompt = 风格提示词前缀 + 本图具体画面 → 交给 zenmux-image-generation 出图
```

`<skill-dir>` = 含本 skill `SKILL.md` 的目录（`功能模块PRD.md` 在其 `references/` 下，即上一级）。
脚本是 TypeScript，用 `npx --yes tsx <脚本>` 运行（本机需有 Node.js/npm，`npx` 会按需拉起 tsx/prettier）。

**先写初稿，再定图单。** 在动手出图之前，先把 PRD 初稿写完（第 1/2/3 章 + 4.1 关键交互流程说明 +
第 5–9 章，4.2 原型图先留空）——初稿里已经把状态、流程分支、异常情况想清楚了，据此反推本功能要画
哪几张原型图（对应关键界面/状态），一次性把所有画面描述列出来让用户确认，再统一生成，减少来回调用
次数。哪张图该画什么、该突出什么状态，都应该能从初稿内容里直接推导出来，不是凌空另起一套画面清单；
初稿里没讲清楚、也推不出来的细节，先照 `澄清问卷.md` 的方法问用户。

### 第 1 步 · 加载平台与风格（预处理 + 缓存）

先跑 `plan`（带 `--project-dir`，指向用户项目根）。首次运行它会把 skill 的配置模板复制到
`<project-dir>/.context/vibe-product-design/config/`，再解析目标平台与当前主题、检查缓存，告诉你
风格提示词是否已就绪：

```bash
npx --yes tsx <skill-dir>/scripts/style_prompt.ts plan --project-dir <project-dir>
```

看返回 JSON 的 `status`：

- **`status: ready`** → 已有缓存的风格提示词（默认的 `apple` 开箱即带，直接就绪）。
  `prompt_prefix` 就是要拼在每张图前面的风格前缀，`rendering` 是出图参数（其中 `size` 已按配置的
  `platform` 定死）。直接进第 2 步。
- **`status: need_generation`** → 该主题还没展开过（用户换了新主题）。返回里的 `brief` 是一份
  **展开任务书**：按它把这句主题**用大模型展开成一段风格提示词**（配色/字体/留白/组件/氛围/规避，
  并内化通用质量底线），写入 `brief` 指定的 `style_file`（即 `.context/.../config/generated/<slug>.style.md`）。
  写完这一步即完成预处理——之后同一主题会一直命中缓存，不必重复展开。写完再跑一次 `load` 取回：

  ```bash
  npx --yes tsx <skill-dir>/scripts/style_prompt.ts load --project-dir <project-dir>
  ```

> 换风格：让用户改 `.context/vibe-product-design/config/prototype-style.yaml` 的 `active_theme`
> ——写成 `themes:` 里的某个 key，或直接写一句自由主题文本（如"赛博朋克霓虹，暗色高对比"）。改完重跑
> `plan` 即按新主题展开。想强制重新展开（改了主题描述但 slug 没变时）加 `--force`。
> 换端：改同一文件的 `platform`（`platforms:` 里的某个 key，如 `web` / `mobile` / `tablet`）——
> 这是项目级设置，改完对后续所有页面级原型图生效；不要在单次产出里私自切换平台。改
> `<skill-dir>/config/` 里的模板没用——运行时以 `.context` 下的副本为准。零第三方依赖：脚本内置极简
> YAML 解析器，`npx` 会按需拉起 tsx，开箱即用。

### 第 2 步 · 逐图拼 prompt 并出图

**每张图的 prompt = 第 1 步拿到的 `prompt_prefix`（风格提示词）＋ 本图的具体画面描述。**
风格由前缀统一保证；画幅/设备**统一按配置的 `platform` 写**（`rendering.platform_label` 就是它的
可读名，如"PC / 桌面 Web（横版 2K）"），不要逐图另选设备。你只需为每张图写清楚**具体画面**：

> <画幅/设备：按配置平台写，如"桌面 Web 界面"（platform: web）或"竖版手机界面"（platform: mobile）>。
> 画面：<具体页面/状态名、关键区域与控件，
> 以及**此刻要突出的关键设计与交互状态**——例如"耳机 App 的降噪设置页：顶部返回箭头与标题
> '降噪模式'，主体是三张竖向排列的选择卡片'关闭 / 通透 / 强降噪'，当前选中'强降噪'卡片用细边框
> 加一枚小圆点标示选中态，卡片下方一行小字标注对应续航影响，页面底部留大量余白">

关键：画面描述里一定要写清**此刻处于哪个状态**、**要突出哪个关键设计**（选中态 / 空状态 / 错误
提示 / 加载中 / 首次引导……），这样每张图才是在"展现设计决策"，而不是画一个通用空界面。同一功能
不同状态各画一张，用状态差异体现设计，不要把所有状态挤进一张图。（这几条通用质量底线已经写进风格
提示词前缀，此处再强调一次，撰写画面描述时对齐。）

**出图参数**用第 1 步 `plan`/`load` 返回里的 `rendering`（来自 `config/prototype-style.yaml`，
所有主题通用，可在配置里改）：`model`（默认 `openai/gpt-image-2`，文字渲染稳、支持自定义尺寸，别用
fast/low 档以免糊字错字）、`quality`（默认 `high`，密集中文必须 high）、`size`（**已由配置的
`platform` 定死**，如 `web` = 2560x1440（PC 2K）、`mobile` = 1024x1536、`tablet` = 2048x1536）。
**`size` 是强约束**：所有页面级原型图一律用它，不要按图自选尺寸；仅组件/局部特写这类非页面级配图
可临时用其它尺寸（如 1024x1024 方形），风格与平台语境仍须一致。具体调用方式、参数含义以
`zenmux-image-generation` 自己的 `SKILL.md` 为准。

### 第 3 步 · 输出路径与数量

生成时**显式指定输出目录**到当前功能的 `assets/` 子目录，不要用 `zenmux-image-generation` 自己的
默认输出路径，否则 PRD 里的相对路径引用会失效：

```bash
--output-dir "<project-dir>/.context/vibe-product-design/<功能 slug>/assets"
```

一个功能通常配 3–8 张图（覆盖主流程每个关键节点 + 1–2 个关键异常/空状态），不凑数、不漏核心
"啊哈"节点或异常提示状态。**每个界面/状态生成 1 张即可**（关键主界面拿不准方向时可生成 2 张对比），
不需要该 skill 默认的 4 变体。生成完成后，在 4.2 节用相对路径 `assets/<文件名>.png` 引用（PRD.md
与 `assets/` 同级，相对路径天然生效，换机器/换用户也不会失效）。

### 前置 · 检查/安装 `zenmux-image-generation`

上面第 2 步出图依赖 `zenmux-image-generation` skill。先检查本机是否已装（常见安装路径如下，装法
不同路径也会不同，找不到就直接跳到安装命令）：

```bash
ls ~/.agents/skills/zenmux-image-generation 2>/dev/null || \
ls <project-dir>/.agents/skills/zenmux-image-generation 2>/dev/null
```

未安装则用 `npx skills add` 安装（`npx` 需要本机已有 Node.js/npm 环境）：

```bash
# 项目级安装（默认，只在当前项目可用）
npx skills add https://github.com/zenmux/skills --skill zenmux-image-generation

# 全局安装（-g，所有项目都能用；只需装一次，之后新项目也不用重复装）
npx skills add https://github.com/zenmux/skills --skill zenmux-image-generation -g
```

装完再跑一遍上面的检查命令确认路径存在即可使用；后续 `zenmux-image-generation` 自身的用法、模型
选择、参数说明，以该 skill 自己的 `SKILL.md` 为准，这里不重复。

## 图文互检的自查清单（出图后必做，这一步产出「设计发现清单」）

画原型图不是给 PRD 配插图，而是一次**设计推演**：把界面具体化时，你被迫替每个元素做出决策——放哪、
怎么写文案、选中态长什么样、出错了显示什么。这些决策里藏着初稿的漏洞和新的灵感。出图后（以及补画后
每一轮），**逐张图对照 PRD 初稿过一遍下面四类问题**，把答案攒成一份「设计发现清单」，再逐条修订进
文档——不要看一条改一条，先攒清单是为了看清发现之间的关联、并把拿不准的一次性问清楚：

| 类型 | 自查问题（逐张图问） | 典型例子 | 修订去处 |
| --- | --- | --- | --- |
| **漏了** | 画这张图时是否出现了初稿没写的状态/元素/分支？空状态、加载中、失败提示、首次引导画出来了吗——初稿有对应描述吗？ | 图里画了"列表为空"的引导文案，但 5.4 异常处理里没有这一条 | 4.1 流程说明、5.2 业务规则、5.4 异常处理、第 7 章测试关注点 |
| **想错了** | 按初稿画出来后，流程/布局是否明显别扭？步骤是否多余、信息层级是否和业务优先级打架、两条规则是否在图上撞车？ | 初稿写"先选模式再看续航影响"，画出来发现续航信息藏太深，应直接标在选项卡上 | 4.1 流程说明、5.2 业务规则，必要时第 6 章验收标准 |
| **新灵感** | 具体化过程中是否冒出更好的交互/更简的流程/值得加的小设计？——**是否纳入本期是产品决策，列出来让用户拍板，不要擅自加范围** | 画选中态时想到"切换模式时给一句续航变化的即时反馈"，初稿没有 | 用户拍板后：纳入 → 相关各章＋图单；不纳入 → 第 9 章开放问题或 1.3 非目标 |
| **不一致** | 图和文字对不上的地方，以谁为准？文案措辞、按钮名称、状态数量、流程顺序，图与 4.1/5.2 是否完全一致？ | 图上按钮叫"开启降噪"，5.2 规则里写的是"启用降噪模式" | 以正确的一方为准，统一改齐（含图注；若图错了且影响理解，列入补画清单） |

清单攒好后：属于产品决策拿不准的（尤其新灵感取舍）先用 `AskUserQuestion` 问用户；然后逐条修订进
上表"修订去处"指向的章节，修订后通读一遍确保前后章节不矛盾。**若修订反过来改变了界面设计**（状态
增删、交互作废、流程节点变化），更新图单、只补画/重画受影响的那几张（风格前缀与出图参数不变），
再回到本清单对一遍。通常一轮回填＋至多一轮补画就该收敛；第二轮后仍大量冒新发现，说明初稿阶段没想
透，停下来和用户重新对齐范围。

## 完成后（设计输入版交付）

图文互检收敛（发现清单已全部修订进文档、图注补全、补画的图也已对检）后，确认 4.2 每张图已用相对
路径链接进文档、图注与正文一致，写入 `<功能 slug>/PRD.md`，运行：

```bash
npx --yes tsx <skill-dir>/scripts/format_feature_prd.ts <project-dir>/.context/vibe-product-design/<功能 slug>/PRD.md
```

规范化格式后，告诉用户文档路径、本次版本号、关键改动，以及原型图数量与存放路径。这份 `PRD.md`
即**设计输入版**交付物——同时提醒用户：拿它和 AI 原型图交给设计师出正式设计稿；设计稿完成后带着
切图回来，走下方「设计稿回填」形成终版 PRD。本次交付到此完成，不要等待设计稿。

## 设计稿回填：从定稿切图到终版 PRD

AI 原型图的使命是把设计意图讲清楚、给设计师一个高质量起点；**设计师的定稿才是产品最终长的样子**。
定稿相对原型几乎必然有出入——视觉重排、文案打磨、控件替换、新增的状态与细节、被砍掉的元素——其中
一部分只是视觉层面的，另一部分**实质上改变了业务规则或边界**。设计稿回填就是把这些出入系统性地
找出来、和用户确认、回填进 PRD，让 `PRD.md` 与设计定稿完全一致，成为下游 Agent（研发/测试/其它
自动化）唯一需要读的**终版设计文档**。

这个环节通常发生在设计输入版交付之后的**另一次会话**：用户说"设计稿好了""切图在 xx 目录""按设计稿
更新 PRD"之类，即进入本环节。前提是该功能文件夹下已有 `PRD.md`；没有就先确认用户是不是指别的功能。

### 第 1 步 · 用户指定切图位置，收进 `design/`

- 问清切图存放位置（本地目录或具体文件），`ls` 验证确有图片；空目录/找不到就回头核对，不要猜。
- `mkdir -p <功能 slug>/design`，把切图**复制**（不要移动用户原始文件）进去，文件名整理成
  `<序号>-<界面/状态名>.png` 的可读形式（如 `01-降噪设置页-默认态.png`）。命名对不上界面时先列
  文件清单让用户对号入座，再复制改名。
- 切图目录里混有与本功能无关的图（设计师常整包导出）时，列清单让用户勾选哪些属于本功能。

### 第 2 步 · 逐张阅读切图，攒「设计定稿差异清单」

用 `Read` 工具逐张读取 `design/` 下的切图（图片可直接阅读），**每张图对照现有 PRD 过一遍下面
四类问题**——方法与「图文互检的自查清单」同源，但对照对象从"初稿 vs AI 原型"换成了"现有 PRD vs
设计定稿"：

| 类型 | 自查问题（逐张切图问） | 典型例子 | 回填去处 |
| --- | --- | --- | --- |
| **新增** | 定稿里出现了 PRD 没写的状态/元素/文案/交互暗示吗？设计师补的空状态、引导、动效示意，PRD 有对应描述吗？ | 定稿在切换降噪模式时加了一条续航变化的即时提示，PRD 没有 | 4.1 流程说明、4.2 图注、5.2 业务规则，必要时第 6 章功能清单加条目 |
| **改动** | 布局、流程顺序、控件形态、按钮/字段措辞与 PRD 一致吗？信息层级变了吗？ | PRD 写三张竖排卡片，定稿改成了分段控件（segmented control） | 4.1、4.2 图注按定稿重写；措辞统一到 5.2/第 6 章 |
| **删减** | PRD 写了、定稿里没有的东西——是被设计师砍了，还是切图没覆盖到？ | PRD 写了"恢复默认"按钮，所有切图里都找不到 | 先问用户是砍了还是漏切；砍了→相关章节移除并记入 1.3 非目标或第 9 章 |
| **影响业务规则** | 上面的差异里，哪些不只是视觉，而是**行为/边界变了**？状态数量、流转触发条件、异常提示时机变了吗？ | 定稿把"确认弹窗"改成了"可撤销的 toast"，撤销窗口期是新的业务规则 | 5.2 业务规则、5.4 异常处理、第 6 章验收标准、第 7 章测试关注点 |

- 逐张看完先**攒成完整清单**再动笔改——和图文互检一样，先看全貌才能发现差异之间的关联。
- 清单里属于产品决策拿不准的，用 `AskUserQuestion` 让用户拍板——尤其两类：**定稿与 PRD 冲突以谁
  为准**（多数时候以定稿为准，但也可能是设计师理解偏了，不要默认任何一方永远正确）；**设计师新增
  的交互是否纳入本期范围**（纳入→回填各章；不纳入→记入第 9 章开放问题）。
- 切图看不清、对应哪个状态拿不准，问用户，不要看图编故事。

### 第 3 步 · 回填全文，切换 4.2 到定稿切图

- 把确认后的差异逐条回填到上表"回填去处"指向的**所有**章节，不是只改第 4 章。
- **4.2 节切换为以 `design/` 切图为准**：每张定稿切图按原有图注结构（这是什么/关键元素/关键设计
  决策/交互说明）写完整图注，相对路径 `design/<文件名>.png` 引用；原 AI 原型图的引用与图注整体
  挪进 4.2 末尾「早期 AI 原型（存档）」小节，加一句"仅作过程存档，以设计定稿为准"。**`assets/`
  目录、图片、原图注一律保留**——中间产物留存，不删。
- 4.3 节按定稿更新：定稿实际复用/新增了哪些组件与模式。
- 回填后**通读全文**：不允许任何章节还在描述已被定稿推翻的旧方案；正文、图注、验收标准里的措辞
  （按钮名、状态名）与定稿完全一致。

### 第 4 步 · 版本、状态与规范化

- 变更记录追加一行：版本 +1，关键改动写明"设计稿回填：以定稿切图（N 张）更新第 4 章及相关业务
  规则/验收标准，形成终版"。
- 文档信息的**状态**改为"已评审"（或用户认可的终版表述）。
- 再跑一次 `format_feature_prd.ts` 规范化。
- 完成后告诉用户：终版 PRD 路径、版本号、回填的切图数量、采纳了哪些关键差异、哪些进了开放问题。
  此后下游 Agent 直接读这份 `PRD.md`，即可获得与设计定稿一致的完整设计。
