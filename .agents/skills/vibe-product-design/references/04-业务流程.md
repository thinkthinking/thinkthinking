# 第 4 章 · 业务流程

> **产出文件：** `04-业务流程.md`
> **输入：** 第 03 章（用户旅程的关键触点）。
> **作用：** 把用户旅程里的关键环节，落成系统/角色层面的端到端流程。它拆出的**流程步骤**，是第 05 章
> 功能设计要支撑的对象——先有流程，再有功能。

## 本章目标

画出关键业务流程的跨角色/跨系统流转，**包含异常路径**（错误、拒绝、超时、回滚），讲清楚每个角色在
每一步该做什么、系统对用户表现出什么行为——这是业务规则和角色协作方式，不是技术实现方案。

## 写作要求

- **必画异常路径。** 快乐路径谁都会画，真正的设计质量在失败分支上。每个核心流程都问"这一步失败了
  会怎样？"——并把回滚/重试/降级处理画出来。
- **从第 03 章的关键触点出发**选流程，优先画核心场景，再画支撑和异常场景。
- 流程要符合真实业务规则，标清参与方、数据流转、权限规则。

## 输出结构

```markdown
# 第 4 章 · 业务流程

## 4.1 流程总览
列出本产品的关键流程清单，分三类：
- **核心流程：** 高频、关键的主流程（如下单支付、内容发布）
- **支撑流程：** 管理、配置、运维类
- **异常流程：** 容错、降级、回滚

## 4.2 核心流程详解
对每个核心流程：

### [流程名]
（Mermaid 时序图或泳道图，见 输出格式规范.md，**必含 alt/else 异常分支**）

- **参与方：** 操作角色 / 相关系统 / 外部接口
- **触发条件：** …
- **主流程步骤：** 1) … 2) … 3) …
- **业务规则：** 处理规则 / 校验规则 / 权限规则
- **信息流转：** 这一步产生/用到什么信息，传递给谁（讲业务含义，不展开存储结构等技术实现）
- **异常处理：** 每条失败分支怎么处理（重试 / 回滚 / 提示 / 降级）

## 4.3 关键异常流程
单独把最重要的异常场景（如支付失败、并发冲突、数据回滚）画清楚。
```
