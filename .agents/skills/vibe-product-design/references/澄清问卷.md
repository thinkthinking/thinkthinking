# 第 2 步 · 问卷澄清指南

流水线第 2 步。用户给了原始想法后，你要先把**影响后续文档走向**的不确定点问清楚，再动笔。
目的只有一个：让最终 PRD 准确反映用户真正想做的东西，而不是你猜的东西。

## 什么该问，什么不该问

**该问**（答案会改变文档内容的关键岔路）：
- 目标用户到底是谁——如果想法里说"帮大家提效"，必须收敛到一类具体的人。
- 核心场景 / 最痛的那个点——决定第 01 章和功能优先级。
- 商业模式方向——免费/订阅/交易抽成？决定第 02 章。
- 平台形态——App / Web / 小程序 / 插件？决定流程和功能。
- 范围边界——这一版要做到哪、明确不做什么？决定 MVP 和功能清单。
- 已有的约束——团队规模、时间、预算、技术栈、已上线的部分（审计场景）。

**不该问**（能自己查到/推断，或属于实现细节）：
- 项目代码库、已有文档、`git log`、配置文件里能查到答案的——去查，别问（见下方「怎么问」）。
- 具体字段、按钮文案、配色这类执行细节。
- 你完全可以基于常识下判断的东西——下判断、标 `[假设]`，别拿去烦用户。
- 一次性把 20 个问题倒给用户。

## 怎么问

- **能探索出来的，别问用户。** 动问之前先想一下：这个不确定点能不能从项目代码、已有文档、
  `git log`、配置文件里自己查到？能查到就去查（用 `Explore`/`Grep`/`Read` 等工具），把查到的结论
  向用户一句话确认即可，别把探索的活儿甩给用户回答。只有真正只有用户脑子里才有答案的（目标用户
  是谁、要不要做某个商业决策、这一版做不做某个范围），才需要问。
- **有依赖关系的问题，一个一个问，问完再问下一个。** 如果答案 A 决定了要不要问、或者怎么问下一个
  问题（比如"目标用户是谁"决定了"核心痛点该往哪个方向问"），就先用 `AskUserQuestion` 单独问第一
  个、等用户回答后再问下一个——不要把有先后依赖的问题一次性都堆给用户，那样只会让人不知道该先想
  哪个。**真正互相独立、谁先答都不影响别的问题**的，才可以合并成一批（同一批最多 2–4 个）。
- 每个问题给 **2–4 个具体选项**，第一个放你推荐的并标"(推荐)"，附一句理由——不管是单独问还是
  批量问，每个问题都要有你的推荐答案，让用户选比让用户写省力。
- 问题要短、具体、互斥。围绕"这个答案会让文档往哪个方向走"来设计。
- 用户答完一轮，**用一两句话复述确认**关键结论，再决定是问下一个依赖问题，还是进入第 3 步逐章产出。

## 特殊情况

- **用户说"你看着办 / 别问直接写"**：不再追问，用合理判断推进，把每个关键假设标进对应章节的
  `[假设]`，并在最终交付时一句话提示"这些地方我替你做了假设，可随时调整"。
- **用户给的材料已经很完整（审计/完善场景）**：只问材料里真正缺的、或自相矛盾的点，不要明知故问。
- **越往后越发现前面问漏了**：可以再补一轮简短问卷，但别打断逐章产出的节奏——能推断的就先推断。

## 两个好问卷的样子（示例）

### 示例一：三个互相独立的问题 → 合并成一批问

> 用户输入："我想做一个帮助自由职业者管理项目和收款的工具。"

这三点互不依赖、谁先答都不影响别的问题，可以合并成一批一次问完：

1. **目标用户更细分**：设计师/开发者/撰稿人…？（影响场景和功能）
2. **最核心的痛**：是"项目散乱难追踪"还是"收款难催款慢"？（决定 MVP 聚焦哪个）
3. **商业模式方向**：订阅制 / 收款抽成 / 免费增值？（决定第 02 章商业画布）

问清这三个，整份 PRD 的骨架就稳了。其余细节边写边用合理判断补，标好假设即可。

### 示例二：有依赖关系的问题 → 一个一个问

> 用户输入："帮我在现有的记账 App 里加个提醒还款的功能。"

先去 `git log`、项目里的 README/现有 PRD 找答案：这个记账 App 是不是已经有"账单"或"负债"相关的
数据模型？如果已经能查到，直接一句话跟用户确认"我看到项目里已有 XX 数据结构，这次提醒功能是基于
它扩展，对吗"，不用再问。

如果查不到、必须问用户，且问题之间有先后依赖，就拆开一个个问，不要一次甩出来：

1. 先问："提醒的触发时机，是按固定日期（如每月 1 号）还是按用户自己设的还款日？"（推荐：按用户
   自己设的还款日，更贴合记账场景）
2. **等用户回答后**，如果答案是"按用户自己设的还款日"，才追问第 2 题："如果用户当天没记这笔账，
   提醒要不要连续弹几天，直到用户处理？"（推荐：连续提醒 3 天后自动降级为一次性摘要，避免打扰）
   —— 这一题的问法本身就取决于第 1 题的答案，所以必须等第 1 题答完才问。

问完一个追问下一个，比一次甩两个问题让用户去消化两套逻辑要清楚得多。
