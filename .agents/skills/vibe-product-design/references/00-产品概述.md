# 第 0 章 · 产品概述

> **产出文件：** `00-产品概述.md`
> **输入：** 用户的原始想法 + 第 2 步问卷澄清的结论。
> **作用：** 全文档的定调章。后面所有章节都建立在这里确立的"真问题 + 目标用户方向 + 范围"之上。

## 本章目标

用一页纸说清楚：**这是个什么产品、解决谁的什么问题、为什么现在做、做成什么样算成功。** 这是 PRD 的
开篇，读者（开发、设计、老板、投资人）看完这一章就该明白产品的本质和野心。

## 写作要求

- 简洁有力，不堆砌。这一章是定调，不是展开——展开在后面各章。
- 真问题要写到"谁会因为它睡不着觉"的程度，不要停在泛泛的行业现象。
- 不许编市场数据；要写就给自底向上的估算逻辑并标 `[假设]`。

## 输出结构

```markdown
# 第 0 章 · 产品概述

## 0.1 一句话定位
为 [目标用户] 在 [场景/需求] 下，提供 [产品形态]，帮助他们 [核心价值]，区别于 [现有替代方案]。

## 0.2 产品背景与真问题
- **真问题：** 我们到底在解决谁的什么问题？他们今天怎么忍受这个问题？
- **为什么是现在：** 什么变化（技术/行为/政策/成本）让这件事现在值得做？
- **问题严重性：** 这个痛有多强、多频繁？（这里只点题，详见第 01 章）

## 0.3 目标与愿景
- **短期目标（MVP 阶段）：** 验证什么、达到什么
- **中长期愿景：** 如果做成，1–3 年后是什么样

## 0.4 产品形态与范围
- **形态：** App / Web / 小程序 / 插件 / …
- **本版范围：** 这一版要做什么
- **明确不做：** 这一版刻意不做什么（聚焦，避免范围蔓延）

## 0.5 核心假设（待验证）
列出这个产品成立必须为真、但目前还没被证实的关键假设，每条标清楚。这是后续各章验证计划的总纲。
- 假设 1：…… `[假设]`
- 假设 2：…… `[假设]`
```

## 完成后

写入 `00-产品概述.md`。在聊天里只说一句"第 0 章产品概述已完成"。进入第 01 章 需求分析。
