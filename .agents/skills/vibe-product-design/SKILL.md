# Vibe 产品设计 —— 从想法到一份清晰可交付的 PRD

你的任务很明确：把用户的**产品想法 / 需求**，变成**一份清晰、可直接交付、用户拿了就喜欢的 PRD
文档**。这个 skill 支持两种规模，先分诊、再选择走哪条路：

```text
理解与分诊   读懂需求本质 → 查产出目录现状 → 问卷确认走哪条路
                │
                ├─ 模式 A · 全量产品设计   从0到1的新产品/新方向，六章工作流逐章产出 → 合并 PRD
                │
                └─ 模式 B · 功能模块 PRD   基于已有产品的单个功能点，问卷澄清 → 按配置风格配高保真原型图 → 单文档产出
                                          （设计师完成设计稿后：指定切图位置 → 阅读切图回填差异 → 终版 PRD）
```

**呈现给用户的最终结果，是一份干净完整的 PRD（模式 A）或一份功能模块 PRD（模式 B）。** 不要给用户
看任何内部脚手架（没有覆盖度地图、没有决策日志、没有状态仪表盘这类东西）。中间的工作产物（分章
文件、草稿）不是交付物，最终的 `PRD.md` 才是。

## 核心原则（两种模式通用）

- **文档要详细，不要偷懒。** 用户要的是能直接用的文档，该写的每一处都要写满、写实、写具体。
  "提升用户体验"是废话；"新用户首次打开后 90 秒内必须完成一次成功操作，否则留存断崖"才是内容。
- **不许编造，存疑先联网。** 凡涉及外部事实——市场规模、竞品现状与定价、行业数据、政策法规、技术/
  价格基准、最佳实践等——只要你**不确定或可能已过时**，先用 `WebSearch`（必要时配 `WebFetch`）联网
  查最新信息，再据实编写、注明来源。**确实查不到真实数据时**，才给出自底向上的估算逻辑并标 `[假设]`，
  绝不凭记忆编造看起来很专业的假数字。（详见下方「联网核实」）
- **超长就分段续写。** 单份文档内容若超出一次输出上限，分多次写，用 `第N段/共M段` 标记，并在每段
  开头用一句话承接上一段，保证 context 连贯（见下方「分段续写规则」）。
- **不要每次都默认走全量流程。** 大多数日常需求是在已有产品上加/改功能——这种时候走模式 B，快、
  聚焦、不啰嗦；只有真正从 0 到 1 的新产品/新方向才走模式 A 的六章分析。
- **PRD 只讲产品需求，不做技术方案设计。** 这份文档要讲清楚的是：产品要做什么、为什么做、给谁用、
  业务规则和产品行为是什么——不负责设计数据库表结构、接口协议、系统技术选型这类软件架构决策。文档
  里的时序图、状态机等图表，画的是**业务流程和状态流转**（一步一步谁做了什么、系统对用户表现出什么
  行为），不是技术实现方案；技术复杂度等评级只是给产品和技术负责人对齐排期用的粗略参考，不要求你去
  设计具体技术方案。写跑偏了、开始讨论用什么数据库/接口协议/技术栈，就是该收回来的信号。

`<skill-dir>` 是包含本 `SKILL.md` 的目录。`<project-dir>` 是用户当前项目根目录——优先用 Git 仓库根
（`git rev-parse --show-toplevel`），否则当前工作目录。

**skill 自带的脚本与配置模板**（在 `<skill-dir>` 下，跟着 skill 走，**skill 升级时会被覆盖**，
所以不要在这里改配置）：

```text

│   ├── prototype-style.yaml          # 原型图配置：目标平台 platform（默认 web/PC 2K，强约束尺寸）
```

**用户项目里的运行时数据都在 `<project-dir>/.context/vibe-product-design/` 这一个产出根下**——
config、模式 A、模式 B 全都归在这里，不再另开 `.context/requirements/`。**配置从 skill 模板复制到
这里后就以这里的为准**——这样 skill 升级覆盖自己的 `config/` 时，不会动到用户的配置和缓存：

```text
<project-dir>/.context/vibe-product-design/      # 本 skill 唯一的产出根（config + 各需求文件夹）
├── .gitignore                                   # 首次写入时自动生成：忽略原型图等大文件，PRD Markdown 照常提交
├── config/                                      # 运行时配置（首次由 style_prompt.ts 从模板复制而来）
│   ├── prototype-style.yaml                     #   用户在这里改风格主题；skill 升级不覆盖
│   └── generated/<slug>.style.md                #   预处理生成/缓存的风格提示词
├── <需求 slug>/                                 # 模式 A：每个产品各占一个文件夹
│   ├── 00-产品概述.md … 05-功能设计.md           #   分章文档
│   ├── _changelog.md                            #   变更记录
│   └── PRD.md                                   #   合并后的最终交付物
└── <功能 slug>/                                 # 模式 B：每个功能各占一个文件夹（与模式 A 平级）
    ├── PRD.md                                   #   单文档交付物（设计稿回填后即「终版 PRD」，下游 Agent 只认它）
    ├── assets/                                  #   AI 高保真原型图（中间产物，回填后也留存，被 .gitignore 忽略）
    └── design/                                  #   设计师定稿切图（设计稿回填环节复制进来，被 .gitignore 忽略）
```

**每一次需求，无论模式 A 还是模式 B，都在产出根下新建一个「以需求命名的文件夹」来存放，不要把文件
散落在产出根本身。** `config/` 与 `.gitignore` 是根下仅有的两个共享项，其余每个直接子文件夹都对应
一个具体需求（一个产品 / 一个功能），两种模式的需求文件夹互为平级、结构自洽。

> **配置 bootstrap**：`style_prompt.ts` 首次运行（带 `--project-dir`）时，会把 `<skill-dir>/config/`
> 复制到 `<project-dir>/.context/vibe-product-design/config/`——**只在目标缺失时复制，绝不覆盖
> 用户已有改动**。之后所有风格配置与缓存的读写都在这里，与 skill 自身解耦。

> **`.gitignore` 自动生成**：三个脚本（`style_prompt.ts` / `merge_prd.ts` / `format_feature_prd.ts`）
> 任一首次写入产出根时，会在 `<project-dir>/.context/vibe-product-design/.gitignore` 落一份忽略规则
> （缺失才写、不覆盖用户改动）：忽略各需求文件夹下的 `assets/` 原型图、`design/` 设计定稿切图与
> `*.png/jpg/...` 等大型二进制，保留 `PRD.md` / 分章 / `_changelog` 等 Markdown 可随仓库提交。
> **本源仓库不应签入任何 `.context/`。**

第一次写入前先建**该需求的文件夹**（`<slug>` 从需求名提炼；模式 B 额外建 `assets/` 子目录）：

```bash
# 模式 A（新建一个产品的产出文件夹）
mkdir -p <project-dir>/.context/vibe-product-design/<需求 slug>
# 模式 B（新建一个功能的产出文件夹，含原型图目录）
mkdir -p <project-dir>/.context/vibe-product-design/<功能 slug>/assets
```

如果文件工具不可用，说明产出本应写在哪里，并把内容留在对话里，不要假装写了文件。

## 理解与分诊（动手写之前，先做这一步）

**动手写任何文档之前，先做三件事：**

1. **读懂需求本质**：用一两句话把你理解的需求复述给用户，确认你抓对了方向。不要一上来甩长问卷。
   匹配用户的语言（主要受众是中文，默认中文）。
2. **查产出根现状**，判断这次更可能是哪种场景（列出产出根下已有哪些需求文件夹）：

   ```bash
   ls -a <project-dir>/.context/vibe-product-design/ 2>/dev/null
   ```

   - 已有需求文件夹（`config/` 之外的子目录，里面有 `PRD.md`/分章文件）→ 说明这个产品/这些功能
     已经被分析过，这次大概率是增量；必要时 `ls` 进相关文件夹看它是模式 A（有分章）还是模式 B。
   - 只有 `config/`（或目录全空）→ 说明这可能是全新产品，也可能只是用户还没跑过这个 skill（不代表
     产品本身是新的）。
3. **用 `AskUserQuestion` 问一次路由问题**，确认走哪条路——除非用户的原始表述已经**明确**说了要
   哪一种（例如直接说"帮我做个功能模块 PRD"或"我要从 0 开始梳理这个新产品"）。问法示例：

   - **选项 A（模式 B · 功能模块 PRD，多数场景推荐）：** 基于已有产品，只为这一个功能点写清楚
     设计、逻辑、边界，配高保真原型图（平台与风格由配置驱动，默认 PC 端 2K ＋ 苹果设计风格），写得快、够聚焦，适合日常迭代/新增功能。
   - **选项 B（模式 A · 全量产品设计）：** 从 0 到 1 做完整分析——需求分析、商业画布、用户旅程、
     业务流程、功能设计六章全套，适合全新产品或全新方向，工作量更大。

   如果用户说"你看着办"：按上一步查到的目录现状推断（已有需求文件夹/全量 PRD → 推断模式 B 的增量；
   只有 `config/` 或全空、且描述听起来是全新方向 → 推断模式 A），**一句话说明推断依据**，给用户一次
   否决机会，而不是直接开始产出。

4. **分诊结果落地**：
   - → **模式 A**：进入下方「模式 A · 全量产品设计」，走原有六章流水线。
   - → **模式 B**：进入下方「模式 B · 功能模块 PRD」。
   - → **设计稿回填**（特例，不必走路由问卷）：用户带着**已完成的设计稿/切图**回来（"设计稿好了"、
     "切图在 xx 目录"、"按设计稿更新 PRD"），且产出根下已有对应功能的模式 B 文件夹（含 `PRD.md`）
     → 直接进入「模式 B · 执行步骤」的**第 8 步 · 设计稿回填**；找不到对应文件夹就先和用户确认是
     哪个功能，不要猜。

## 模式 A · 全量产品设计

适合从 0 到 1 的新产品或新方向。这是一条**线性流水线**，严格按步骤走，不要跳步、不要自创花活
（第 0 步先查已有 PRD）：

```text
第 0 步 · 查已有     先看产出目录是否已有 PRD —— 有就在其上修改、并记一笔 changelog
第 1 步 · 接收输入   用户给出前期产品想法或原始需求
第 2 步 · 问卷澄清   你读懂后，对不确定的关键点用问卷工具和用户探讨清楚
第 3 步 · 逐章产出   按六个章节工作流依次输出详细文档（超出输出限制就分多次续写）
第 4 步 · 合并 PRD   用脚本把各章节文档合并成一份完整的 PRD.md（含开头的变更记录表）
```

### 章节顺序即逻辑

章节顺序是从用户视角逐步走到要造的产品功能：先搞清用户是谁要什么，再看生意成不成立，再看用户怎么
体验、业务怎么运转，最后才落到造什么功能。后面的章节直接引用前面的结论，**绝不重述**。

### 第 0 步 · 先查已有 PRD（增量优先）

**动手写之前，先列一下产出根**，看看这个产品是否已有对应的需求文件夹，判断这是「从 0 新建」还是
「在已有 PRD 上修改」：

```bash
ls -a <project-dir>/.context/vibe-product-design/ 2>/dev/null            # 看有哪些需求文件夹
ls -a <project-dir>/.context/vibe-product-design/<需求 slug>/ 2>/dev/null # 命中疑似文件夹后再看里面
```

- **已有匹配的需求文件夹、里面有 `PRD.md` 或分章文件（`00-产品概述.md` 等）** → 这是一次**修改**。
  先读相关分章文件（以及 `_changelog.md`），在**现有内容基础上增改**，不要推倒重写、不要新建一套
  覆盖掉旧的。改完务必更新 `_changelog.md`（见下方「变更记录（Changelog）」），版本号 +1。
- **没有匹配文件夹** → 这是一次**新建**：先从产品名提炼一个 `<需求 slug>`，`mkdir -p
  <project-dir>/.context/vibe-product-design/<需求 slug>` 建好该产品的文件夹，正常走第 1→4 步，并在
  写第一章之前先在该文件夹里创建 `_changelog.md`，写入首条 `v1.0` 记录。
- 全篇下文所说的「产出目录」，都指这个**该需求专属的文件夹** `<project-dir>/.context/vibe-product-design/<需求 slug>/`，
  不是产出根本身——不要把分章文件散落在产出根下。

无论新建还是修改，**`_changelog.md` 都要在跑合并脚本之前就位**——脚本会把它嵌进最终 PRD 的开头。

### 六个章节 = 六个工作流

这是流水线第 3 步的执行核心。**每个章节对应一个工作流文件，产出一个分章文档。** 处理某章时，
**先读它的工作流文件**，再忠实执行其中的输出规范——不要凭记忆简化。

| 序 | 分章文档 | 工作流文件 | 内容要点 |
| --- | --- | --- | --- |
| 0 | `00-产品概述.md` | `references/00-产品概述.md` | 一句话定位、真问题、为什么是现在、目标与愿景、核心假设 |
| 1 | `01-需求分析.md` | `references/01-需求分析.md` | 目标用户/画像/痛点(强度×频率)/JTBD/竞品/机会点/产品定位 |
| 2 | `02-商业画布.md` | `references/02-商业画布.md` | 精益画布九宫格 + 各模块验证计划（问题/客户引用第01章，不重述）|
| 3 | `03-用户旅程.md` | `references/03-用户旅程.md` | 用户旅程图 + 旅程分析说明书，标注啊哈时刻与情绪低点 |
| 4 | `04-业务流程.md` | `references/04-业务流程.md` | 跨角色泳道/时序图 + 流程说明（**含异常路径**）|
| 5 | `05-功能设计.md` | `references/05-功能设计.md` | 功能架构图 + L1–L4 功能清单(优先级/工作量/复杂度) + MVP + 迭代路线 |

> **去冗余说明：** 原始工作流里"用户需求分析"与"精益画布"的问题/客户/竞品有重叠——本设计让画布
> 引用需求分析、不重述。原"功能需求分析"与"功能优先级分析"已**合并为第 5 章功能设计**，优先级、
> 版本、迭代一次到位。风险与验证**融入各章**（画布的验证计划、旅程的痛点、流程的异常路径），
> 不单开章节。

```text
00 这是什么 → 01 用户是谁、要什么 → 02 生意成不成立 → 03 用户怎么体验 → 04 系统怎么运转 → 05 要造什么功能
```

旅程(03)、流程(04) 放在功能设计(05) 之前，是因为功能是用来**支撑**旅程触点和流程步骤的——先有
体验和流程，功能设计就能直接引用它们，既符合逻辑又消除冗余。

### 变更记录（Changelog）

每份 PRD 开头都要有一张**变更记录表**，记录每次编辑的日期、版本、维护人、关键改动。它单独存为
产出目录下的 `_changelog.md`（下划线开头，不计为章节），合并脚本会自动把它嵌到最终 `PRD.md` 的
开头（标题之后、目录之前）。

#### 维护人怎么取

维护人一栏写**两类署名**：发起这次编辑的人 + 你这个 AI 智能体。

- **人类维护者**：从当前项目的 git 配置取 name 和 email：

  ```bash
  git config user.name && git config user.email
  ```

  写成 `姓名 <邮箱>` 的形式（如 `程知微 <zhiwei.cheng@example.com>`）。取不到时留 `—` 或问用户。
  **这是从当前项目 git 配置里读到的真实署名，不要用示例里的占位符替换成别的假名字。**
- **AI 智能体**：写你自己的名字 `Claude (vibe-product-design)`。

所以维护人一栏形如：`程知微 <zhiwei.cheng@example.com> ＋ Claude (vibe-product-design)`。

#### 表格格式

`_changelog.md` 的内容就是一段以二级标题开头的表格，**最新的记录放在最上面**：

```markdown
## 变更记录

| 日期 | 版本 | 维护人 | 关键改动 |
|---|---|---|---|
| 2026-06-20 | v1.1 | 程知微 <zhiwei.cheng@example.com> ＋ Claude (vibe-product-design) | 重写第 05 章功能清单，新增 MVP 迭代路线 |
| 2026-06-18 | v1.0 | 程知微 <zhiwei.cheng@example.com> ＋ Claude (vibe-product-design) | 初始版本，产出全部六章 |
```

- **日期**用环境里的「今天日期」，格式 `YYYY-MM-DD`。
- **版本**用语义化的 `v主.次`：大改（重做某章/结构调整）进主版本，小改/补充进次版本；首版 `v1.0`。
- **关键改动**写**这次实际改了什么**，具体到章节与要点，别写"优化文档"这种废话。
- 新建 PRD 时建首条 `v1.0`；每次修改在最上面**追加一行**，不要改写历史行。

### 流水线四步详解

#### 第 0 步 · 先查已有 PRD

见上方「第 0 步 · 先查已有 PRD（增量优先）」：先 `ls` 产出目录，决定新建还是修改，并确保
`_changelog.md` 就位。

#### 第 1 步 · 接收输入

读完用户给的原始想法 / 需求。用一两句话把产品本质复述给用户，确认你抓对了方向（这一步在「理解与
分诊」阶段已经做过，此处不必重复）。

#### 第 2 步 · 问卷澄清

按 `references/澄清问卷.md` 的方法，挑出**真正影响后续文档走向**的不确定点，用 `AskUserQuestion`
工具和用户探讨清楚。原则：

- 只问会改变文档内容的关键问题，能从代码库/已有文档里探索到答案的先去探索，不问能自己合理推断
  的细节。
- **有依赖关系的问题一个一个问、答完一个再问下一个**；只有真正互相独立的问题才合并成一批（一批
  最多 2–4 个）。每个问题给出推荐选项 + 理由（见「怎么问」）。
- 用户答完一轮，把结论简短复述确认，再决定是问下一个依赖问题，还是进入逐章产出。
- 如果用户说"你看着办 / 别问那么多直接写"，就用合理判断推进，把假设标进对应章节。

#### 第 3 步 · 逐章产出

按 0→1→2→3→4→5 的顺序，**一章一章**地产出。每章：

1. 读对应的工作流文件，按其输出规范写**详细**内容。
2. **写之前先核实外部事实**：本章若要用到市场规模、竞品/定价、行业数据、政策、技术或价格基准等
   外部信息，而你不确定或它可能已过时，先用 `WebSearch` 联网查最新信息再动笔（详见「联网核实」）。
3. 写入对应分章文件（如 `02-商业画布.md`）。
4. 引用前面章节已确立的结论，不重述。
5. 写完一章在聊天里只说一句"第 X 章已完成，写入 xx.md"，不整段重放内容。
6. 内容超长时按下方规则分段续写。

##### 分段续写规则（保证 context 连贯）

当某一章内容预计超过单次输出上限时：

- 把该章拆成 `00-产品概述.md`、`00-产品概述.part2.md`… 这样的续写文件（主文件不带 part 后缀，
  续写从 `.part2` 起）。`merge_prd.ts` 会按 `part` 序号自动接续合并。
- **每个续写段开头写一行** `<!-- 续写自 00-产品概述.md 第N段/共M段 -->`，并用一句话承接上一段
  结尾（"接上一段的功能清单，继续 L3 功能点…"），确保跨段连贯。
- 不要为了省事而压缩内容——宁可多分一段，也要把该写的写完整。

#### 第 4 步 · 合并 PRD

所有章节产出完成后，**先确认 `_changelog.md` 已更新到本次版本**（见「变更记录」），再运行合并脚本
生成最终交付物：

```bash
npx --yes tsx <skill-dir>/scripts/merge_prd.ts <project-dir>/.context/vibe-product-design/<需求 slug>/
```

脚本会：先用 **prettier** 把每个分章文件（含 `_changelog.md`）就地规范化，再按章节序号（及 part
序号）拼成一份带目录的 `PRD.md`、把变更记录表嵌在开头，最后对整份 PRD 再过一遍 prettier，输出在同一
目录下。完成后告诉用户最终 PRD 的路径、本次版本号与关键改动，并简要说明它包含哪几章。这份 `PRD.md`
就是呈现给用户的、清晰完整、且**格式合规**的最终成果。

**Markdown 规范化（prettier）** —— 脚本通过 `npx --yes prettier`（GFM 表格）自动修正格式：标题
前后空行、列表缩进、表格对齐、去行尾空白、合并多余空行、文件末尾单换行等。用 `--prose-wrap preserve`
不重排中文段落换行，Mermaid 等代码块内容、`[假设]` 标记、续写注释都原样保留。

- 无需预装：`npx --yes` 会按需拉起 prettier；本机需有 Node.js/npm 环境（跑 TS 脚本本来也需要）。
- 无法调用 prettier（无网络/无 npx）时脚本会**跳过规范化并提示**，合并照常完成。
- 调试或不想改动分章文件时，可加 `--no-lint` 跳过规范化。

## 模式 B · 功能模块 PRD

适合**基于已有产品的增量功能设计/改进**——大多数日常需求都属于这类。不需要重新分析用户画像、
商业模式、竞品，只需要把**这一个功能点**的设计、逻辑、边界写清楚到"造火箭说明书"级别，让 UI/UX
设计、研发、测试、PM 各自拿着这一份文档就能开工。

### 目录与命名

```text
<project-dir>/.context/vibe-product-design/
└── <功能名 slug>/     # 每个功能各占一个文件夹，直接放在产出根下（与模式 A 的产品文件夹平级）
    ├── PRD.md          # 单文档交付物——始终只维护这一份；设计稿回填后它就是「终版 PRD」
    ├── assets/         # AI 高保真原型图（平台/风格由 config 驱动，默认 PC 2K ＋ 苹果风），PRD.md 用相对路径 assets/xxx.png 引用
    └── design/         # 设计师定稿切图（第 8 步「设计稿回填」时从用户指定位置复制进来），PRD.md 用相对路径 design/xxx.png 引用
```

- `<功能名 slug>` 从功能名提炼一个简短目录名（去空格/特殊符号），如给已有耳机 App 加"降噪模式
  切换"功能，目录名就是 `降噪模式切换`。
- **先查产出根**，判断是否已有匹配的功能文件夹：

  ```bash
  ls -a <project-dir>/.context/vibe-product-design/ 2>/dev/null
  ```

  - 已有匹配文件夹 → 这是一次**修改**，在现有 `PRD.md` 上增改，更新其中的变更记录表、版本号 +1。
  - 没有 → 在产出根下新建该功能的文件夹和它的 `assets/` 子目录。
  - **拿不准是否匹配已有文件夹时，跟用户确认一句，不要静默猜测覆盖**——这是唯一一份交付物，覆盖错
    代价很高。
- 模式 B 只有一份文件，**不需要**单独的 `_changelog.md`、**不需要**跑合并脚本；变更记录作为
  `PRD.md` 内的一个章节直接维护（维护人取法同模式 A，见上方「维护人怎么取」）。

### 执行步骤

**核心节奏是「PRD ⇄ 原型图 的螺旋迭代」**：先把业务逻辑想清楚、写成初稿文字，再据此画原型图。
画原型图本身就是一次深度设计推演——把界面具体化的过程会暴露初稿没想到的状态、边界、交互分支，也会
带来新的灵感。所以图出来后不是"贴图收工"，而是带着这些发现回头**修订 PRD**；若修订反过来改变了
界面设计，再补画/重画受影响的图——如此交替一到两轮，直到图文互相印证、再无实质新发现。这正是产品
经理的真实工作方式：文档和原型互为草稿、互相打磨，最终交付的是一对"完美咬合"的 PRD ＋ 原型图。
全程遇到查不到、推不出的不确定点，都照 `references/澄清问卷.md` 的方法用 `AskUserQuestion` 问用户，
不要凌空猜测业务规则或交互细节。

**模式 B 的交付分两个阶段**：第 1–7 步产出「设计输入版 PRD ＋ AI 原型图」，交给设计师去出正式
设计稿——到这里第一次交付即完成，**不要卡在这里等设计稿**；等设计师完成后，用户带着定稿切图回来，
再走第 8 步「设计稿回填」——阅读设计定稿、找出它相对原型阶段的所有细节与改动、回填进 PRD——产出
**终版 PRD**。始终只维护 `PRD.md` 这一份文档（没有 `PRD-final.md` 之类的副本），设计稿回填后它就是
下游 Agent（研发/测试/其它自动化）唯一需要读的完整设计文档；`assets/` 里的 AI 原型图是中间产物，
留存不删。

1. **查已有**（见上）：确认是新建还是修改；若产出根下已有相关产品的模式 A 文件夹（含 `PRD.md`/分章
   文件），读取作为现有产品的背景信息，不要重新问用户已经写过的内容。
2. **轻量问卷澄清**：比模式 A 更聚焦，按 `references/澄清问卷.md` 的方法问清楚下面四点。这四点有
   先后依赖——前一点的答案决定了后一点该怎么问——**按顺序一个一个问，答完一个再问下一个**，不要
   一批甩出来：
   1. 这个功能建立在哪个现有产品/模块之上？**先查**产出根下是否已有相关产品的模式 A 全量 PRD
      文件夹、或其它功能的模式 B 文件夹，能查到就直接引用、一句话跟用户确认，不必问。
   2. 这次要解决的核心问题/触发原因是什么？（答案会决定下一步该往哪个方向问影响范围）
   3. 影响范围：基于上一题的核心问题，涉及哪些现有页面/角色/系统？
   4. 边界：这次明确不做什么？（等前面范围定下来之后再问，否则用户不知道该排除什么）
   用户说"你看着办"时，用合理判断推进并把假设标进对应章节。
3. **撰写 PRD 初稿（先把业务逻辑写清楚，原型图先留空）**：按 `references/功能模块PRD.md` 的模板，
   先写好第 1/2/3 章（背景目标、范围约束、用户与场景）、第 4.1 节关键交互流程说明（文字描述状态/
   步骤，不画图）、第 5–9 章（业务逻辑、功能清单与验收标准、测试关注点、数据与埋点、风险）。
   把状态流转、异常分支、边界值都想清楚、写具体——这是后面出图和验收的依据。4.2 高保真原型图这一节
   先留空（后面据初稿出图后再补）。**写的过程中一旦碰到查不到、推不出的不确定点**（业务规则、边界
   条件、交互细节），停下来照 `references/澄清问卷.md` 的方法问用户，不要自己拍板编。
4. **依据初稿确定原型图清单**：初稿里第 4.1 节和第 5 章已经把状态、流程分支、异常情况写清楚了——
   基于这些反推需要画哪几张图（覆盖主流程关键节点 + 关键异常/空状态），一次性列给用户确认，而不是
   凌空想象要画什么。若某张图的画面细节初稿里没讲清楚、也推不出来，同样先问用户。
5. **生成高保真原型图（平台与风格由配置驱动）**：按 `references/功能模块PRD.md` 的「原型图生成规范」
   一节执行——这是一条 **config 驱动**的流水线：
   - **平台与风格都不写死**：先跑 `style_prompt.ts plan`（带 `--project-dir`）。首次运行它会把
     skill 的配置模板复制到 `<project-dir>/.context/vibe-product-design/config/`（升级不覆盖），并
     读取其中 `prototype-style.yaml` 的 `platform`（目标平台，默认 web/PC 2K）和 `active_theme`
     （用户写的一句话风格主题，默认苹果设计风格）。

     ```bash
     npx --yes tsx <skill-dir>/scripts/style_prompt.ts plan --project-dir <project-dir>
     ```

     若返回 `status: ready` 直接拿到缓存的风格提示词；若 `status: need_generation`，按返回的
     `brief` 把这句主题**用大模型展开**成一段风格提示词，写入 `brief` 指定的 `style_file`
     （`.context/.../config/generated/<slug>.style.md`，一次性预处理，之后命中缓存），再 `load` 取回。
   - **逐图出图**：每张图 prompt = 风格提示词前缀 ＋ 本图具体画面；检查/安装 `zenmux-image-generation`，
     按第 4 步确认过的清单出图，出图参数用 `plan`/`load` 返回的 `rendering`。**其中 `size` 由配置的
     `platform` 决定，是强约束**：所有页面级原型图一律按该平台的分辨率与界面形态出，不要按图自选
     尺寸、更不要混出别的端的图；仅组件/局部特写这类非页面级配图可临时用其它尺寸（如 1024x1024）。
   - 产出的是能**展现关键设计决策**的高保真界面稿（真实文案、按状态各出一张），不是灰框线框图。
     生成时**显式指定输出目录**到本功能的 `assets/` 子目录，**不要用该 skill 自己的默认输出路径**，
     也不要用它默认的 4 变体（每个界面/状态 1 张，关键主界面拿不准方向时给 2 张对比即可）。
6. **图文互检 · 螺旋修订 PRD**（这是模式 B 的核心价值环节，不要退化成"贴图＋补图注"）：

   1. **先攒一份「设计发现清单」再动笔改**：逐张审视原型图，同时回想出图过程中你写画面描述时被迫
      做出的每一个具体化决策（元素摆放、文案措辞、状态标示、信息层级……），对照 PRD 初稿逐项自问——
      按 `references/功能模块PRD.md`「图文互检的自查清单」过一遍：哪些是初稿**漏了**的（新状态、
      新边界、缺失的异常分支）？哪些是初稿**想错了**的（画出来才发现流程别扭、规则冲突）？哪些是
      画图时冒出的**新灵感**（更好的交互、更简的流程、值得加的小设计）？哪些是图和初稿**不一致**
      的（以谁为准）？把发现逐条列出来，属于产品决策拿不准的（尤其"新灵感"是否纳入本期）用
      `AskUserQuestion` 让用户拍板，不要自己猜、也不要擅自扩大范围。
   2. **把清单逐条修订进 PRD**：确定要采纳的发现，回填到**对应的所有章节**——4.1 流程说明、
      5.2 业务规则、5.4 异常处理、第 6 章功能清单与验收标准、第 7 章测试关注点，必要时连 1.3
      非目标、2.1 影响范围、第 9 章风险也一并更新——不是只改第 4 章。修订后通读一遍，确保新内容
      与既有章节没有互相矛盾。
   3. **补全图注、链接进文档**：把 4.2 节每张图的图注（这是什么/关键元素/关键设计决策/交互说明）
      写完整，用相对路径 `assets/<文件名>.png` 把图链接进 PRD。图注要与修订后的正文一致。
   4. **判断是否需要再转半圈**：若第 2 步的修订**反过来改变了界面设计**（新增了要展现的状态、
      某张图画的交互已经作废、流程节点增删），就更新图单、只补画/重画**受影响的那几张**（沿用第 5
      步的风格前缀与参数），然后回到本步第 1 小步再对一遍；若修订只动了文字层面（规则细化、验收
      补充），图无需重画，螺旋收敛。通常一轮回填 ＋ 至多一轮补图就该收敛——第二轮之后仍冒出大量
      新发现，说明初稿阶段想得不够，该停下来和用户重新对齐范围，而不是无限转圈。

   这几轮写作是**同一次产出内部的节奏**，完成后只算一次交付、只在变更记录里留一条记录（新建 v1.0 /
   修改版本 +1），不需要在版本历史里体现"草稿版"或每轮螺旋。
7. **格式规范化**：

   ```bash
   npx --yes tsx <skill-dir>/scripts/format_feature_prd.ts <project-dir>/.context/vibe-product-design/<功能 slug>/PRD.md
   ```

   完成后告诉用户文档路径、本次版本号、关键改动，以及原型图数量与存放路径。这份 `PRD.md` 即
   **设计输入版**交付物——同时告诉用户：拿它和 AI 原型图给设计师出正式设计稿，**设计稿完成后带着
   切图回来**，走下面第 8 步把设计定稿回填进 PRD、形成终版。本次会话到此为止，不要等待。
8. **设计稿回填 · 形成终版 PRD**（独立的后置环节，通常在设计师完成设计稿后的**新会话**里触发——
   用户说"设计稿好了"、"按设计稿更新 PRD"、给出切图路径等，都进入这一步）：

   1. **让用户指定设计稿切图位置**：用 `AskUserQuestion` 或直接对话问清切图存放在哪里（本地目录/
      文件路径）。拿到路径后 `ls` 确认里面确有图片文件；空目录或找不到就回头和用户核对，不要猜。
   2. **把切图收进本功能的 `design/` 目录**：`mkdir -p <功能 slug>/design` 后把用户切图**复制**
      （不要移动）进去，保持/整理成可读的文件名（如 `01-降噪设置页-默认态.png`）。这样 PRD 的相对
      路径引用不因用户挪动原始目录而失效。切图数量很多时先列清单给用户过目，确认哪些属于本功能。
   3. **逐张阅读设计稿，攒「设计定稿差异清单」**：用 `Read` 逐张读取切图（它是图片，Read 工具可以
      直接看），对照现有 PRD（尤其 4.1 流程说明、4.2 原型图注、5.2 业务规则、5.4 异常处理、第 6 章
      验收标准）逐项自问——方法与第 6 步图文互检同源，见 `references/功能模块PRD.md`「设计稿回填：
      从定稿切图到终版 PRD」一节：设计稿**新增**了什么（新状态、新元素、新文案、新交互暗示）？
      **改动**了什么（布局、流程顺序、控件形态、措辞与 PRD 不一致）？**删减**了什么（PRD 写了但
      设计稿没有）？哪些改动**影响业务规则**（不只是视觉，而是行为/边界变了）？把发现逐条列出。
      属于产品决策拿不准的（尤其"设计稿与 PRD 冲突时以谁为准"、设计新增交互是否纳入本期）用
      `AskUserQuestion` 让用户拍板，不要默认设计稿永远正确、也不要默认 PRD 不可动。
   4. **把差异清单逐条回填进 PRD**：确认后的差异回填到对应的**所有**章节——重点重写第 4 章：
      4.1 流程说明按定稿更新；**4.2 节改为以 `design/` 切图为准**——每张定稿切图按原有图注结构
      （这是什么/关键元素/关键设计决策/交互说明）重写图注，用相对路径 `design/<文件名>.png` 引用；
      原 AI 原型图的引用整体挪进 4.2 末尾新增的小节「4.2.x 早期 AI 原型（存档）」，一句话说明
      "以下为设计定稿前的 AI 原型，仅作过程存档，以上方设计定稿为准"，**不删除 `assets/` 目录和
      原图注内容**。行为层面的差异同步更新 5.2 业务规则、5.4 异常处理、第 6 章验收标准、第 7 章
      测试关注点等。修订后通读全文，确保没有任何章节还在描述已被设计稿推翻的旧方案。
   5. **更新变更记录并再次格式规范化**：变更记录追加一行（版本 +1，关键改动写明"设计稿回填：
      以定稿切图更新第 4 章及相关业务规则，形成终版"），文档信息里的**状态**改为"已评审/终版"，
      再跑一次 `format_feature_prd.ts`。完成后告诉用户：终版 PRD 路径、本次版本号、回填了多少张
      设计稿切图、有哪些关键差异被采纳。此后下游 Agent 直接读这份 `PRD.md` 即可获得与设计定稿
      一致的完整设计。

## 联网核实（拿不准就搜，别凭记忆）

两种模式通用。PRD 的可信度取决于事实的准确度。**只要某个外部事实你不确定、或它会随时间变化，就先
联网查最新信息，再据实编写。** 宁可多搜一次，也不要凭记忆写一个可能过时或错误的"专业数字"。

### 什么时候必须搜（`WebSearch`，必要时 `WebFetch` 取详情）

- **市场规模 / 行业数据**：用户量、市场容量、增长率、渗透率、客单价水平等。
- **竞品现状**：竞品是否还存在、最新功能、定价方案、融资/市占情况——这些最容易过时。
- **定价与单位经济基准**：对标产品的价格点、行业毛利、获客成本、第三方服务（如大模型 API、云、
  短信、支付）的最新报价。
- **政策 / 法规 / 合规**：行业准入、数据与隐私规定、面向特定人群（如未成年人）的监管要求等。
- **技术可行性与基准**：某能力当前是否成熟、主流方案、性能/成本基准。

### 什么时候不必搜

- 能靠常识或自底向上逻辑合理推断的（标 `[假设]` 即可）。
- 纯属用户内部信息、产品主观设计决策（这些问用户，不是搜网）。
- 不影响文档结论的细枝末节。

### 怎么用

- 当月时间以环境里的「今天日期」为准，检索"最新 / 2025 / 2026"等限定，确保拿到现行信息。
- 多个独立事实可并行发起多次搜索；信息冲突时以更权威、更新的来源为准。
- **查到了**：据实写入，并在该处标注来源（如"据 XX 2026 报告"或附链接），不要再标 `[假设]`。
- **没查到可靠数据**：退回自底向上估算逻辑并标 `[假设]`，在文中一句话说明"未检索到公开数据，下为估算"。
- 联网工具不可用时：明确说明该数据无法核实、按 `[假设]` 处理，不要假装搜过。

## 输出工艺

两种模式通用（编码体系略有差异，见下）。组装内容时遵循 `references/输出格式规范.md`：P0–P3
优先级、XXS–XXL 工作量、技术复杂度评级、Mermaid 图（架构/泳道/旅程）、精益画布画法。**精益画布等用
Mermaid 或 Markdown 表格，不要画 SVG。** 功能编码上，模式 A 用完整的 L1–L4 四级体系；模式 B 只是
单个功能点，不套四级，编码规则见 `references/功能模块PRD.md` 第 6 章。

写的时候**用规范的 Markdown**（标题前后留空行、表格用标准 `| --- |` 语法、代码块用三反引号围栏），
最终细节交给合并脚本（模式 A）或格式化脚本（模式 B）里的 prettier 统一规范化——你不必手抠对齐，
但结构要写对。

## 不要做的事

- 不要给用户看内部脚手架（覆盖度地图、决策日志、状态仪表盘）——最终只交付一份 PRD。
- 不要每次都默认走模式 A 的全量六章分析——大多数需求是增量功能，先分诊，多数场景走模式 B。
- 不要跳过「理解与分诊」就闷头开始产出——除非用户的原始表述已经明确说了要哪种模式。
- 不要偷懒压缩文档；该详细的地方要详细，超长就分段续写。
- 不要在文档里重述已经写过/已经引用过的内容——引用即可。
- 不要凭记忆写可能过时的外部事实；拿不准就先 `WebSearch` 联网核实（见「联网核实」）。
- 不要编造市场数据、竞品事实、用户数；联网仍查不到才标 `[假设]` 并给估算逻辑。
- 不要把 PRD 写成技术方案/系统设计文档。时序图、状态机、数据与埋点等图表和章节，描述的都是**业务
  规则和产品行为**（谁在什么条件下做了什么、系统对用户表现出什么状态），不是数据库表结构、接口协议、
  技术选型——发现自己在设计这些，就是该收回到产品视角的信号。
- 不要跳过模式 A 第 2 步问卷澄清、或模式 B 的轻量问卷就闷头写（除非用户明确要你直接写）。
- 不要把有先后依赖的问题一次性甩给用户；能从代码库/已有文档探索到的答案不要问用户（见
  `references/澄清问卷.md`「怎么问」）。
- 不要在模式 B 写初稿、定原型图清单、或图文互检的任何一个环节里凌空猜测业务规则或交互细节——查不到、
  推不出的不确定点，随时照 `references/澄清问卷.md` 的方法用 `AskUserQuestion` 补问用户，不要为了
  推进流程自己拍板编。
- 不要把模式 B 的图文互检做成"贴图＋补图注"就收工——先攒「设计发现清单」（漏了什么/想错什么/新灵感/
  图文不一致），逐条修订进对应的所有章节；修订改变了界面设计就补画/重画受影响的图，直到图文互相
  印证。也不要反向失控：新灵感是否纳入本期要用户拍板，螺旋通常一轮回填＋至多一轮补图就该收敛。
- 不要把模式 B 第 8 步「设计稿回填」做成"换图收工"——必须逐张阅读设计定稿切图、攒「设计定稿差异
  清单」（新增/改动/删减/影响业务规则的），回填进第 4 章**和**受影响的业务规则、验收标准、测试
  关注点等章节；设计稿与 PRD 冲突以谁为准要用户拍板，不要默认任何一方永远正确。
- 设计稿回填时不要删除中间产物：`assets/` 下的 AI 原型图和原图注保留（图注挪进"早期 AI 原型
  （存档）"小节），只是 4.2 的正式引用切换为 `design/` 下的定稿切图。也不要另建 `PRD-final.md`
  之类的副本——终版就是同一份 `PRD.md`，版本号 +1、状态改为终版，下游 Agent 只认它。
- 设计稿回填时不要凭空猜测切图位置或含义——切图路径让用户指定并 `ls` 验证；切图对应哪个界面/状态
  拿不准就问用户，不要看图编故事。
- 不要把文件散落在产出根：每次需求（模式 A 或 B）都在 `.context/vibe-product-design/` 下新建一个
  「以需求命名的文件夹」存放，产出根本身只放 `config/` 和 `.gitignore`。
- 不要跳过查已有产出这一步：动手前先 `ls` 产出根，看有没有匹配的需求文件夹，有就在其上改、别推倒
  重写、别静默猜测覆盖。
- 不要漏掉变更记录：模式 A 每次产出/修改都要更新 `_changelog.md`；模式 B 直接在 `PRD.md` 内的
  变更记录章节更新（日期、版本+1、维护人、关键改动）。
- 不要自己手写 Markdown lint 规则去抠格式——规范化交给脚本里的 prettier（模式 A 用 `merge_prd.ts`，
  模式 B 用 `format_feature_prd.ts`，都用 `npx --yes tsx` 运行），别绕过。
- 不要改 `<skill-dir>/config/` 里的配置模板——那是模板，skill 升级会覆盖。配置以
  `<project-dir>/.context/vibe-product-design/config/` 下的为准（首次运行由脚本从模板复制过去）。
- 模式 B 的原型图必须生成到本功能文件夹的 `<功能 slug>/assets/` 目录下——不要用
  `zenmux-image-generation` 自己的默认输出路径，否则 PRD 里的相对路径引用会失效。
- 不要把原型图风格写死或凭记忆瞎编——风格由 `.context` 下 `prototype-style.yaml` 的 `active_theme`
  驱动，出图前先跑 `style_prompt.ts plan --project-dir <项目根>` 拿到（或按 `brief` 预处理生成）
  对应的风格提示词。
- 不要违背配置的目标平台——`prototype-style.yaml` 的 `platform`（默认 web/PC 2K）是强约束：页面级
  原型图的分辨率、画幅、界面形态（布局范式/导航结构）一律按它出，不要按图自选尺寸、不要混出别的端
  的图；仅组件/局部特写可临时用其它尺寸。用户要换端就改配置里的 `platform`，不是在单次产出里切换。
- 不要让用户手填一堆配色/字体/留白细节——用户只写一句话主题，细节由预处理用大模型展开成风格
  提示词文件并缓存复用。
- 不要产出灰框占位 / 手绘线框图——始终是高保真、含真实文案、能展现关键设计决策的界面稿；
  不同状态各出一张。
- 不要自创流程或章节；模式 A 严格按六章和四步走，模式 B 严格按 `references/功能模块PRD.md` 的
  模板走。
